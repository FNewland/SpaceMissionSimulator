3. NEW CLASSES AND INTERFACES
================================================================================

3.1 FaultPropagator
    Purpose: Manages fault detection and cascading effects
    Key Methods:
    - register_rule(rule: FaultPropagationRule)
    - detect_fault(fault_id, current_time) -> list[dict]
    - clear_fault(fault_id)
    - active_faults() -> list[str]
    - get_recovery_state(fault_id) -> RecoveryState
    - increment_recovery_attempt(fault_id, current_time) -> bool
    Usage: Tracks which faults are active, applies cascade rules

3.2 LoadSheddingManager
    Purpose: Manages priority-based load shedding by battery SoC
    Key Methods:
    - register_stage_config(stage: int, config: dict)
    - get_required_stage(soc: float) -> int
    - update_stage(soc: float, current_time: float) -> Optional[int]
    - execute_stage(stage: int)
    - current_stage() -> int
    Usage: Automatically reduces loads as battery depletes

3.3 ProcedureExecutor
    Purpose: Manages sequential procedure execution
    Key Methods:
    - register_procedure(proc_id, config)
    - register_command_callback(cmd_name, callback)
    - start_procedure(proc_id, current_time) -> str
    - tick_procedures(current_time)
    Usage: Executes S8 command sequences automatically

3.4 RecoveryState
    Purpose: Tracks recovery attempts for each fault
    Fields:
    - fault_id, response_taken, timestamp
    - recovery_attempts, max_attempts
    - last_attempt_time, state (IDLE/RUNNING/COMPLETE/FAILED)
    Usage: Implements escalation logic

3.5 FaultPropagationRule
    Purpose: Defines how faults cascade
    Fields:
    - fault_id, description, primary_param, threshold
    - cascades: list[dict] with target subsystem, action, delay
    Usage: Loaded from YAML, used by FaultPropagator

3.6 ProcedureExecution
    Purpose: Tracks state of executing procedure
    Fields:
    - exec_id, proc_id, steps, start_time
    - state (RUNNING/COMPLETE/FAILED)
    - current_step, completed_steps
    Usage: Maintained by ProcedureExecutor

6. LOAD SHEDDING THRESHOLDS
================================================================================

Stage Transitions (Battery SoC):
- Stage 0 (Normal):    SoC >= 30%
- Stage 1 (Reduced):   SoC 20-30%
- Stage 2 (Minimal):   SoC 10-20%
- Stage 3 (Survival):  SoC < 10%

Stage Actions:
- Stage 0: All loads operational
- Stage 1: Payload standby, heaters 50%, aux off
- Stage 2: Payload off, AOCS safe, heaters off
- Stage 3: Only OBC, TTC RX, battery heater 10%, AOCS safe

Implementation: Checked every tick in _tick_fdir_advanced()

8. RECOVERY ESCALATION
================================================================================

Level 1: Parameter-level responses (disable wheel, reduce heater)
Level 2: Subsystem safe mode (AOCS/EPS/OBC safe mode)
Level 3: Spacecraft emergency (all non-essential off)

Escalation Timing:
- Level 1 → Level 2: After 30 seconds if unresolved
- Level 2 → Level 3: After 60 seconds if unresolved
- Level 3: Final state, no further escalation

Tracking: RecoveryState.recovery_attempts <= max_attempts (3)

================================================================================
9. PROCEDURE EXECUTION FLOW
================================================================================

Step Execution:
1. Procedure starts at T=t0
2. For each step:
   - Wait for step delay (absolute time from start)
   - Execute registered callback
   - Mark step as complete
3. When all steps complete:
   - Emit PROCEDURE_COMPLETED event (0x0F08)
   - Remove from active procedures
4. On any callback error:
   - Emit PROCEDURE_FAILED event (0x0F09)
   - Halt procedure (don't continue to next step)

Callbacks: S8 commands routed through dispatcher

11. KEY DESIGN DECISIONS
================================================================================

1. Load Shedding by SoC Threshold
   - Simple, deterministic, based on battery state only
   - No hysteresis between stages (direct threshold)
   - Checked every tick for responsiveness

2. Procedure as YAML + Step Callbacks
   - Flexible, extensible, human-readable
   - No special procedure language needed
   - Can invoke any registered callback

3. Cascading via Separate Rule List
   - Independent of traditional FDIR rules
   - Can enable/disable without affecting parameter monitoring
   - Clear separation of concerns

4. Recovery State per Fault
   - Tracks attempts independent of rules
   - Supports escalation logic
   - Allows manual recovery restart

5. Events for All Major Actions
   - FAULT_DETECTED on any violation
   - RECOVERY_STARTED on response
   - LOAD_SHED_* on stage changes
   - PROCEDURE_* on procedure lifecycle
