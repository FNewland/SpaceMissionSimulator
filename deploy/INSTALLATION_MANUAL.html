1. System Overview

The SMO (Space Mission Operations) Suite is a complete spacecraft operations simulation and training system for the EOSAT-1 ocean monitoring mission. It consists of five components that can run on a single machine or be distributed across a network:

GatewayTM/TC relay for multi-site MCS deployments10025 (TCP)

2. Prerequisites

Required

  • Python 3.11 or newer (3.12, 3.13 also work)

  • Internet access during installation (to download Python packages)

3. Quick Start (Single Machine)

To get everything running on one machine in under 2 minutes:

# 1. Clone or extract the project
cd SpaceMissionSimulation

# 2. Run the installer
./deploy/install.sh

# 3. Start all services
./deploy/start-all.sh

# 4. Open in your browser:
#    MCS:        http://localhost:9090
#    Planner:    http://localhost:9091
#    Instructor: http://localhost:8080
That's it! For single-machine use with default settings, no configuration changes are needed. The default smo-env.conf uses localhost for everything.

4. Network Architecture

4.1 Single Machine

All five components run on the same machine. The MCS connects directly to the simulator's TCP ports. Browsers on the same machine (or network) access the web UIs.

Machine A (your computer): Simulator → TC :8001, TM :8002, HTTP :8080 MCS → HTTP :9090 (connects to localhost:8001/8002) Planner → HTTP :9091 Browser access: http://localhost:9090 ← MCS http://localhost:9091 ← Planner (standard) http://localhost:9091/wide ← Planner (5760×1080) http://localhost:8080 ← Instructor

4.2 Two-Machine Split (Simulator + Ops)

Simulator on one machine, MCS and Planner on another. Common for training where the instructor machine is separate from operators.

Machine A (Instructor): 192.168.1.100 Simulator → TC :8001, TM :8002, HTTP :8080 Machine B (Operators): 192.168.1.101 MCS → HTTP :9090 (connects to 192.168.1.100:8001/8002) Planner → HTTP :9091 Browsers on network: http://192.168.1.101:9090 ← MCS http://192.168.1.101:9091 ← Planner http://192.168.1.100:8080 ← Instructor

4.3 Full Distribution (3+ Machines)

Machine A (Simulator): 192.168.1.100 Simulator → TC :8001, TM :8002, HTTP :8080 Machine B (MCS): 192.168.1.101 MCS → HTTP :9090 Machine C (Planner): 192.168.1.102 Planner → HTTP :9091 Machine D (Gateway): 192.168.1.103 [optional] Gateway → :10025 (relays to Simulator)

4.4 Multi-Site with Gateway

The Gateway allows multiple MCS instances at different locations to share a single simulator. It fans out TM to all connected clients and aggregates TC upstream.

Example: Simulator on Machine A, MCS+Planner on Machine B

Machine B (MCS + Planner) — 192.168.1.101

#    SIMULATOR_HOST="192.168.1.100"
#    PLANNER_REACHABLE_HOST="192.168.1.101"

# 3. Start MCS and Planner
./deploy/start-mcs.sh &

SIMULATOR_TM_PORT=8002
SIMULATOR_HTTP_PORT=8080

MCS_HTTP_PORT=9090
PLANNER_HTTP_PORT=9091
PLANNER_REACHABLE_HOST="192.168.1.101"  # Machine B's own IP

Example: Full 3-Machine Setup

Machine A — Simulator (192.168.1.100)

./deploy/install.sh --sim-only
./deploy/start-simulator.sh

Machine B — MCS (192.168.1.101)

# smo-env.conf:
SIMULATOR_HOST="192.168.1.100"
PLANNER_REACHABLE_HOST="192.168.1.102"

./deploy/start-mcs.sh

Machine C — Planner (192.168.1.102)

8. Gateway (Multi-Site MCS)

The Gateway is a transparent TM/TC relay that allows multiple MCS instances to connect to a single simulator. Use it when you have operators at multiple locations.

Setup

# On the gateway machine — edit deploy/smo-env.conf:
USE_GATEWAY="yes"
GATEWAY_HOST="192.168.1.103"
GATEWAY_LISTEN_PORT=10025
GATEWAY_UPSTREAM_HOST="192.168.1.100"   # Simulator
GATEWAY_UPSTREAM_PORT=8002              # Simulator TM port

# Start the gateway:
./deploy/start-gateway.sh

MCS Configuration to Use Gateway

# On each MCS machine — edit deploy/smo-env.conf:
USE_GATEWAY="yes"
GATEWAY_HOST="192.168.1.103"
GATEWAY_LISTEN_PORT=10025

# The MCS start script will automatically connect to the gateway
# instead of the simulator when USE_GATEWAY="yes"
./deploy/start-mcs.sh

9. Wide-Screen Planner Display (5760 × 1080)

The planner includes a special layout optimized for ultra-wide displays (e.g., triple-monitor setups at 5760 × 1080 resolution). It shows all panels side by side in a 5-column layout:

Accessing the Wide-Screen UI

# Standard planner:
http://<planner-host>:9091

# Wide-screen planner (5760×1080):
http://<planner-host>:9091/wide
Display Setup For the best experience on a 5760×1080 display:
  • Use Chrome or Firefox in full-screen mode (F11)
  • Set browser zoom to 100%
  • The layout uses CSS flexbox and adapts to the available width
  • It also works on standard screens but panels will be narrower

10. Remote Browser Access

All web UIs (MCS, Planner, Instructor) bind to 0.0.0.0 by default, meaning they accept connections from any network interface. Any browser on the same network can connect.

Finding Your IP Address

# macOS
ipconfig getifaddr en0        # Wi-Fi
ipconfig getifaddr en1        # Ethernet

# Linux
hostname -I | awk '{print $1}'
# or
ip addr show | grep "inet " | grep -v 127.0.0.1

# Windows
ipconfig | findstr "IPv4"

Connecting from Other Machines

Replace localhost with the host machine's IP address:

# If MCS is running on 192.168.1.101:
http://192.168.1.101:9090          # MCS
http://192.168.1.101:9090          # (any browser on the network)

# Multiple operators can connect simultaneously
# Each selects their operator position in the MCS UI
Multiple Operators The MCS supports 6 simultaneous operator positions: Captain, EPS, AOCS, OBDH, TTC, and Payload. Each browser session selects a position from the dropdown. Position-based access control restricts which commands each operator can send.

12. Packaging for Distribution

To create a clean distributable archive for deployment on other machines:

# Create the package
./deploy/package.sh

# Or with a custom name
./deploy/package.sh --name eosat1-training

# Output: smo-suite-20260330-143000.tar.gz

What's Included

  • All 5 source packages (smo-common, smo-simulator, smo-mcs, smo-planner, smo-gateway)
  • Mission configuration (configs/eosat1/)
  • All 64 procedures, 28 scenarios, 11 manual sections
  • Deployment scripts and this installation manual
  • Test suite (996 tests)
  • Documentation

What's NOT Included

  • Python virtual environment (.venv) — recreated by install.sh
  • SQLite TM archives (data/*.db)
  • Build artifacts (__pycache__, *.egg-info)

Deploying on a Target Machine

# 1. Copy the archive
scp smo-suite-*.tar.gz user@target-machine:~/

# 2. On the target machine
ssh user@target-machine
tar xzf smo-suite-*.tar.gz
cd smo-suite-*

# 3. Install
./deploy/install.sh

# 4. Configure
vi deploy/smo-env.conf

# 5. Run
./deploy/start-all.sh

13. Component Reference

13.1 Simulator

The spacecraft simulator implements a full PUS-C service stack (Services 1, 3, 5, 6, 8, 9, 11, 12, 15, 17, 19, 20) with pluggable subsystem models for EPS, AOCS, TCS, OBDH, TTC, and Payload.

  • Entry point: smo-simulator --config configs/eosat1/ [--speed 1.0]
  • Instructor UI: Inject failures, run scenarios, set breakpoints, control sim speed
  • Subsystems: 6-panel solar array, dual magnetometers, 6 CSS heads, 2 star cameras, battery-heater thermal, multispectral payload (4 bands)
  • State machines: Separation (7 phases), bootloader, transponder command channel

13.2 MCS (Mission Control System)

WCAG 2.1 AA-compliant web UI with 6 operator positions and 11 subsystem tabs.

  • Entry point: smo-mcs --config configs/eosat1/ --connect <host>:<port> --port 9090
  • Tabs: Overview, EPS, AOCS, TCS, OBDH, TTC, Payload, Commanding, PUS, Procedures, Manual
  • Features: Real-time telemetry via WebSocket, SVG block diagrams, Chart.js trending, alarm journal, procedure execution, GO/NO-GO coordination, world map, stored TM playback
  • Archive: SQLite-backed TM archive for historical parameter queries

13.3 Planner

Mission planning with orbit prediction using SGP4 propagation.

  • Entry point: smo-planner --config configs/eosat1/ --port 9091
  • Standard UI: http://host:9091/ — map + contacts + schedule (stacked)
  • Wide-screen UI: http://host:9091/wide — 5 columns side by side (5760×1080)
  • Features: Contact window prediction, pass-based scheduling, conflict detection, power budget (SoC prediction), data budget (downlink capacity), imaging target planner (7 ocean current targets)

13.4 Gateway

Transparent bidirectional TM/TC relay for multi-site deployments.

  • Entry point: smo-gateway --upstream <host>:<port> --listen 0.0.0.0:10025
  • TM: Fans out to all connected downstream clients
  • TC: Forwards from any downstream client to the upstream simulator

14. Troubleshooting

14.1 "Command not found: smo-simulator"

# Make sure the virtual environment is activated:
source .venv/bin/activate

# Re-install the package:
pip install -e packages/smo-common -e packages/smo-simulator

14.2 "Address already in use"

# Another process is using the port. Find and kill it:
lsof -i :8080    # macOS/Linux
# or
netstat -tlnp | grep 8080   # Linux

# Kill the process:
kill <PID>

# Or change the port in deploy/smo-env.conf

14.3 MCS cannot connect to Simulator

  • Verify the simulator is running: curl http://<sim-host>:8080/api/state
  • Check that SIMULATOR_HOST in smo-env.conf is correct
  • Check firewall rules (ports 8001, 8002 must be open on the simulator machine)
  • Try telnet <sim-host> 8002 to verify TCP connectivity

14.4 Browser shows "WebSocket connection failed"

  • The MCS will fall back to HTTP polling automatically
  • Check that you're using the correct host/port in the browser URL
  • Corporate proxies may block WebSocket — try direct network access

14.5 Planner shows "SIMULATED" instead of "LIVE"

  • The planner needs time (a few seconds) to compute initial contact windows
  • Check the planner log output for errors
  • Verify the TLE data in configs/eosat1/orbit.yaml is valid

14.6 Wide-screen planner doesn't fill the screen

  • Make sure the browser is in full-screen mode (F11)
  • Set browser zoom to 100% (Ctrl+0 / Cmd+0)
  • The layout adapts to available width — it works best at 5760px but scales to other widths

14.7 Python version error

# If you get "requires-python >= 3.11" errors:
python3 --version   # Must be 3.11+

# Install Python 3.11 specifically:
# macOS:  brew install python@3.11
# Ubuntu: sudo apt install python3.11 python3.11-venv

# Then recreate the venv:
rm -rf .venv
python3.11 -m venv .venv
./deploy/install.sh

14.8 Running Tests

source .venv/bin/activate
pip install pytest pytest-asyncio
pytest tests/ -v   # All 996 tests
