1. System Overview & Mission Profile

1.1 Mission Summary

EOSAT-1 is a 6U multispectral imaging CubeSat designed for ocean current monitoring. The spacecraft operates in a 450 km sun-synchronous orbit (98° inclination) with a multispectral camera capturing ocean colour data at four wavelengths: 443, 560, 665, and 865 nm. These bands enable measurement of chlorophyll concentration, suspended sediments, and sea-surface temperature gradients — key indicators for tracking ocean currents, upwelling zones, and coastal water quality.

The mission employs two ground stations providing approximately 6–8 contact windows per day, with a combined daily downlink capacity of roughly 120 MB using S-band communications. The spacecraft carries cold-redundant components for all critical avionics to ensure mission continuity.

Orbit450 km circular SSO, 98° inclination
Period~94 min (15.24 rev/day)

Power6 body-mounted GaAs panels, 120 Wh Li-ion battery
RedundancyCold-redundant: transceiver, PDM, OBC (A/B), magnetometers (A/B)

1.2 Suite Composition

The Space Mission Operations (SMO) Suite is a Python monorepo of five packages that together provide a complete spacecraft operations training and simulation environment. Each package runs as an independent process, communicating via TCP sockets and HTTP/WebSocket — mirroring real ground-segment architectures while remaining deployable on a single laptop for classroom use.

2. Systems Architecture

2.1 High-Level Architecture

The architecture follows a classical ground-segment model divided into three regions: the Spacecraft Simulator, Ground Segment servers, and Operator Interfaces (browser-based). The simulator produces ECSS PUS-C compliant telemetry packets at 1 Hz and accepts telecommands over a dedicated TCP channel, while the ground segment translates these into operator-friendly displays and planning tools.

High-Level Architecture — SMO Suite

2.2 Data Flow Summary

All data flows use the ECSS PUS-C standard layered over CCSDS space packets. Telecommands flow from the operator's browser through WebSocket to the MCS, which builds PUS packets and forwards them over a dedicated TCP channel (port 8001) to the simulator. Telemetry flows in the reverse direction: the simulator generates S3 housekeeping, S5 events, and S1 verification packets at 1 Hz, streaming them over TCP port 8002 to the MCS, which decommutates, archives, limit-checks, and broadcasts results to all connected browsers via WebSocket.

3. Package & Module Structure

3.1 smo-common — Shared Infrastructure

The common library is a dependency of every other package. It provides the ECSS/CCSDS packet codec (build/parse with CRC-16), TCP length-prefix framing for reliable stream demultiplexing, an SGP4-based orbit propagator with eclipse and contact-window computation, 29 Pydantic configuration schemas for YAML validation, and a telemetry builder that assembles housekeeping, event, and verification packets.

3.2 smo-simulator — Spacecraft Simulator

The simulator is the largest package, responsible for modelling the spacecraft's behaviour at 1 Hz. The SimulationEngine orchestrates a deterministic tick loop: propagate the orbit, tick each subsystem model, emit housekeeping packets, check FDIR rules, and manage the mission phase state machine. Six subsystem models — each implementing the SubsystemModel ABC — simulate power, attitude, thermal, on-board data handling, communications, and payload operations with configurable fidelity.

7.3 Command Dispatch Flow

When a telecommand arrives at the simulator, it passes through a multi-stage pipeline that mirrors real flight software behaviour. Phase gating restricts which services are available during early mission phases (e.g., during BOOTLOADER_OPS only S17, S9.1, and S8 functions 42–47 are permitted). Power state checks ensure the target subsystem is actually powered on before accepting commands.

Raw TC bytes
  │
  ├─ decommutate_packet() → PrimaryHeader + SecondaryHeader + data
  │
  ├─ _check_tc_acceptance(service, subtype, data)
  │   ├─ Phase gating (bootloader restricts to S17, S9.1, S8 func 42-47)
  │   ├─ Power state check (is target subsystem powered?)
  │   └─ Returns (accepted: bool, error_code: int)
  │
  ├─ if rejected → S1.2 Acceptance Failure TM → enqueue
  ├─ if accepted → S1.1 Acceptance Success TM → enqueue
  │   │
  │   ├─ S1.3 Execution Start TM → enqueue
  │   │
  │   ├─ ServiceDispatcher.dispatch(service, subtype, data)
  │   │   ├─ Routes to handle_s3/s5/s6/s8/s9/s11/s12/s15/s17/s19/s20
  │   │   ├─ S8 → routes func_id to subsystem.handle_command()
  │   │   └─ Returns list[bytes] (response TM packets)
  │   │
  │   ├─ if success → S1.7 Execution Complete TM → enqueue
  │   └─ if failure → S1.8 Execution Failure TM → enqueue
  │
  └─ All TM → tm_queue → broadcast to MCS

8. Telemetry Processing Pipeline

8.1 TM Generation (Simulator Side)

The simulation engine generates telemetry in a deterministic 1 Hz tick loop. Each tick updates all subsystem models, then checks whether any housekeeping SIDs are due for emission based on their configured intervals. Events are generated by edge detection on subsystem state changes (e.g., mode transitions, threshold crossings). The S12 on-board monitoring service performs per-tick limit checks and generates transition reports when parameters cross defined thresholds.

Engine tick loop (1 Hz)
  │
  ├─ Subsystem.tick() → writes to shared_params[param_id]
  │
  ├─ _emit_hk_packets(dt)
  │   └─ For each enabled HK SID with elapsed interval:
  │       TMBuilder.build_hk_packet(sid, params, hk_structure)
  │       → S3.25 TM packet {SID + packed parameter values}
  │       └─ Enqueue to tm_queue
  │
  ├─ _check_subsystem_events() → edge detection
  │   └─ _emit_event() → TMBuilder.build_event_packet()
  │       → S5.{1-4} TM packet (event_id, severity, CUC, aux_text)
  │       └─ trigger_event_action() → may execute S8 auto-response
  │
  ├─ _tick_s12_monitoring() → ServiceDispatcher.check_monitoring()
  │   └─ Out-of-limit? → S12.9 Transition Report TM
  │
  └─ _enqueue_tm(pkt)
      ├─ if downlink_active: tm_queue.put(pkt)
      └─ always: TMStorage.store_packet(service, pkt)

8.2 TM Processing (MCS Side)

The MCS receives raw ECSS packets over TCP, decommutates them, and routes based on service type. S3 housekeeping packets are decommutated into named parameters, limit-checked against configured thresholds, and pushed to all connected browser clients. S5 events are logged to the alarm journal with severity badges. The TMArchive persists all parameters to SQLite every 10 seconds for historical retrieval.

TCP :8002 → read_framed_packet() → MCSServer._tm_receive_loop()
  │
  ├─ decommutate_packet() → DecommutatedPacket
  │
  ├─ _process_tm(pkt)
  │   ├─ S1: Update _verification_log (accepted/rejected/completed/failed)
  │   ├─ S3.25: TMProcessor._process_hk(data)
  │   │   ├─ Extract SID → lookup hk_structure
  │   │   ├─ Decommutate parameters → _params[id], _history[id]
  │   │   └─ Limit check → alarms if out-of-range
  │   ├─ S5: Extract event → alarm_journal (if severity >= 3)
  │   ├─ S12.9: Monitoring violation → alarm_journal
  │   └─ S15: Playback data → _tm_dump_data
  │
  ├─ TMArchive.store_parameters(params) — every 10 seconds
  │
  └─ WebSocket broadcast → all connected clients
      ├─ {type: "tm", packet: {...}}
      └─ {type: "alarm", alarm: {...}} (if applicable)

9. State Machine Diagrams

9.1 Spacecraft Phase State Machine

The spacecraft progresses through seven mission phases, from pre-separation through to nominal operations. Phase transitions are triggered by timers, automatic conditions, or instructor commands. Each phase gates which PUS services and subsystems are available, faithfully reproducing the real constraints during LEOP and commissioning.

UML State Diagram — Mission Phase Transitions

9.2 AOCS Mode State Machine

The AOCS subsystem implements a 9-mode state machine governing attitude control strategy. Transitions between modes are guarded by conditions (e.g., angular rate thresholds, CSS validity, star tracker status) that must be sustained for a configurable duration before the transition fires. This prevents mode flickering due to transient sensor noise.

UML State Diagram — AOCS Modes

9.3 Procedure Runner State Machine

The ProcedureRunner manages the execution of operator procedures (YAML-defined step sequences). It supports three step types: wait_s (timed delay), wait_for (condition polling), and command (PUS command dispatch). A step-by-step mode pauses after each step, requiring step_advance() for progression — useful for training scenarios.

10. Sequence Diagrams

10.1 Telecommand Execution Sequence

The full telecommand lifecycle — from operator click to verification report — is shown in section 2.2 above. The key insight is that every TC produces at minimum two verification TM packets (S1.1 acceptance, S1.7/S1.8 completion), with optional S1.3 (start) and S1.5 (progress) reports for long-running commands. This full S1 lifecycle is tracked by the MCS's verification log, which displays green/red/blue badges for each command's status.

10.2 HK Telemetry Cycle

Housekeeping telemetry follows a regular 1 Hz cycle. Each subsystem writes its parameters into the shared store during tick(). The TMBuilder then checks which HK SIDs have elapsed their configured interval, packs the relevant parameters into S3.25 packets, and enqueues them for transmission. On the MCS side, these packets are decommutated back into named parameters, limit-checked, and broadcast to all browser clients for chart updates.

15. Concurrency & Thread Safety

15.1 Threading Model

The simulator engine runs in a dedicated daemon thread at 1 Hz, while its HTTP server (instructor interface) runs on the main thread's asyncio event loop. The MCS and planner are fully async (aiohttp). Cross-thread communication uses Python's threading.Lock for the shared parameter store and asyncio.Lock for WebSocket client lists and GO/NO-GO state.
